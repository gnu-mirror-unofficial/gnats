    Introduction to GNATS

    GNU GNATS is a set of tools for tracking bugs reported by users to a central site. It allows problem report management and communication with users via various means. GNATS stores all the information about problem reports in its databases and provides tools for querying, editing, and maintenance of the databases.

    Thanks to its architecture, GNATS is not bound to a single user interface--it can be used via command line, e-mail, Emacs, or a network daemon, usually used with a Web interface. Together with the fact that all GNATS databases and configuration can be stored in plain text files, it allows easy use and provides good flexibility. Basically, if the GNATS tools do not provide everything you need, you can add your own additional utilities using standard GNU tools.

    There is a lot of confusion about the status of GNATS, and version numbering has been somewhat unclear. There was a version 3.2 release in 1993, but GNATS 3.2 is really broken, so don't use it. Work continued on versions numbered 3.1xx, and many major advances have been made to GNATS in the last couple of years, including multiple database support, customized fields, and comprehensive WWW and Tk based front-ends.

    The newest GNATS release is 3.113.1, and the next official release of GNATS will be version 4.0. After a lull in activity during the year 2000, development efforts have picked up again. The 4.0 documentation is nearly done, but some cleanup and bugfixing in the code itself remains. GNATS 4.0 has been through two alpha releases in 1999, but since then a lot of changes have been made, so we probably need another alpha version. A solid beta version is expected in late fall 2001.

    Several third-party interfaces and other tools that work with GNATS have been contributed, the most actively developed are Gnatsweb, a web-based interface, and TkGnats, a Tcl/Tk-based interface for UNIX/Linux and Windows, which can be found in the contrib subdirectory of the GNATS distribution. Releases of these tools aren't necessarily synchronized with GNATS releases, see the section on downloading below for details on how to get the newest version.

    If you'd like to hear about GNATS announcements, subscribe to the info-gnats mailing list.

    Downloading GNATS

    GNATS

    The most recent released versions of GNATS are available from ftp://ftp.gnu.org/pub/gnats/.

    The last "official" version is 3.113. If you consider to create a new GNATS installation, we recommend you to use the current development version of GNATS 4. The new version is slightly incompatible with the old one, it has many new useful features and we believe it is also more stable. Moreover, we provide only a very limited support for GNATS 3.

    The current development version is available via CVS only, see the GNU savannah GNATS project page for the instructions how to get it.

    See information about other GNATS resources for the location of development code. There are also available Debian packages of both released and development versions of GNATS.

    Gnatsweb

    Gnatsweb, a GNATS web interface, development is hosted in a separate CVS repository at gnu.org. Gnatsweb has two development branches, one for version 2 (which works with GNATS 3.x) and one for version 4 (which works with GNATS 4).

    Get the current Gnatsweb 4 development code as follows: 

    cvs -d:pserver:anoncvs@subversions.gnu.org:/cvsroot/gnatsweb login
(press "enter" when prompted for password)
cvs -z3 -d:pserver:anoncvs@subversions.gnu.org:/cvsroot/gnatsweb co gnatsweb

    Get the current Gnatsweb 2 development code as follows:

    cvs -d:pserver:anoncvs@subversions.gnu.org:/cvsroot/gnatsweb login
(press "enter" when prompted for password)
cvs -z3 -d:pserver:anoncvs@subversions.gnu.org:/cvsroot/gnatsweb co -r gnatsweb-v2-branch gnatsweb

    Reporting GNATS Bugs

    Please report GNATS bugs to the GNATS bug tracking system (preferred) or to the e-mail address <bug-gnats@gnu.org>.

    GNATS uses (gasp) GNATS to track problem reports. We've got the Gnatsweb web interface set up so you can browse the database or submit new problem reports there. Use guest as both the login name and the password. You can also try out the Gnatsweb interface if you want to see how things look, but please do not submit reports that aren't real. This database is being used by GNATS developers, and is not meant for testing purposes.

    All PR transactions/additions/changes are recorded on the gnats-prs mailing list.

    Getting involved in GNATS development

    We are always on the lookout for people willing to contribute to the project. We need more active developers. Besides, patches for existing problems, contributions to documentation, answers to frequently asked questions or ideas for future development are all very welcome. And, if you aren't a developer or feel able to write good documentation, there is still something you can do: Simply use GNATS and report any bugs and quirks you may find.

    GNATS

    The current maintainer of GNATS itself is Milan Zamazal. The immediate goal is to get a usable version of GNATS 4 out of the door. This involves making it more robust and updating the documentation.

    Gnatsweb

    The current maintainers of Gnatsweb are Yngve Svendsen and Gerald Pfeifer. Several Gnatsweb 2.x releases have been made in the past 6 months, fixing bugs and improving usability and functionality. The 2.x branch is however purely in maintenance mode now, and no further releases are planned unless bugs of earth-shattering importance surface. Focus is now on version 4 of Gnatsweb, which will work only with GNATS 4. A beta release of Gnatsweb 4 is available as part of the current CVS version of GNATS 4. See the section above on availability of development versions for instructions on how to obtain the latest development snapshot of Gnatsweb 4. Anyone interested in participating in Gnatsweb development either by submitting patches or taking directly part in planning and development should get in touch with Yngve or Gerald. Patches should in general be sent to the help-gnats mailing list.

    How to produce patches

    Even though people may have good coding skills, they may never have submitted patches to projects such as this before. Following is a quick how-to on producing a patch file.

    Check out the GNATS version you intend to modify from the CVS repository, then make the changes you intend to. To produce the patchfile, use the following command: 

    cvs diff -up file1 file2 ... > patchfile.out

    where file1, file2 etc. are the paths to the files that you changed in the source tree. This creates a patch file patchfile.out which contains the differences between the source files in the CVS repository and the checked out files you edited. Since current development efforts are directed at getting version 4 out of the door, patches against the latest development version will be most useful. Patches should go to the help-gnats mailing list.

    Full CVS documentation is available at http://www.loria.fr/~molli/cvs/doc/cvs_toc.html.
