Bob - I've been looking at the V4 source and have a question.

First, it's reasonable for a front-end like TkGnats to be able to, given
ONLY a gnats server hostname and port, present the user with a menu of
all databases on that server. Then, having selected a particular database,
do all the normal stuff that the front-end can do now: submit, query,
view, edit. All this with no knowledge of the database other than the
server and port.

This requires the front-end to:

1) get a list of databases from gnatsd (DBLA).
2) get the GNATS_ADDR (bugs email address) from gnatsd (LCFG).

I've implemented this in my current DEV version of TkGnats, but hit the
problem that LCFG requires VIEW access (something that I had done in the
core myself). I had to do some fancy coding in TkGnats to be able to get
the userid/passwd FIRST in order to do the LCFG command to cover the case
where a server is setup with NONE as the default access. DBLA is not
restricted.

Then I went looking to see how this might all work with V4.

First, I see that DBLA is now part of LIST, and LIST requires VIEW access.
Second, I don't see anywhere that gnatsd will return GNATS_ADDR (the
gnats-adm/config file seems to be gone).

What I'm after is probably that the LIST command be unrestricted (this
requires discussion!) and that the GNATS_ADDR can be retrieved somehow.

Perhaps just "LIST Databases" should be unrestricted (as before), but add
the GNATS_ADDR for each database to the returned list. Hopefully nobody
thinks the bugs email addresses need to be hidden...

Comments please!
